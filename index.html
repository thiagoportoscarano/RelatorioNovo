Nota Metodológica

Aperfeiçoamento dos testes e do sistema

ℹ️
Os testes foram aperfeiçoados entre as etapas. Na Medição 1, os scripts não evidenciavam o ponto exato de deterioração do desempenho — o cenário atingia pico abruptamente e os estágios não eram granulares o suficiente. Para a Medição 2, os cenários foram reescritos com rampa mais granular (estágios de 10 VUs), permitindo identificar em qual faixa de concorrência o sistema começa a degradar. Como os testes foram modificados, ambas as versões do sistema (antes e depois das otimizações) foram re-testadas com os novos scripts, e os resultados da Medição 1 abaixo já refletem a re-execução com os scripts aperfeiçoados. Desta forma, a comparação é válida e homogênea.

Scripts aperfeiçoados — o que mudou

Os novos scripts aumentam a carga de 10 em 10 VUs, com etapas de 15s cada, até 100 VUs (registro) e 80 VUs (login). Isso permite observar exatamente em qual faixa de concorrência a latência ultrapassa o threshold — o "joelho da curva". O sleep(1) foi mantido para simular comportamento humano realista.

Otimizações implementadas no sistema

Entre as duas medições, três otimizações foram aplicadas: (1) remoção das chamadas obsoletas de pool set_core_connections_per_host que causavam crash no startup; (2) redução dos parâmetros do Argon2 (memory_cost de 65 536 → 32 768, time_cost de 3 → 2); (3) simplificação da query de unicidade — remoção do filtro WHERE tipo = ? que exigia ALLOW FILTERING no Cassandra.

Potenciais gargalos identificados na Medição 1

🔴 Gargalo 1 — Custo computacional do Argon2 (ALTO impacto)

O hash Argon2 com memory_cost=65536 e time_cost=3 consome ~300–800ms por operação isolada. Sob 100 VUs simultâneos, todas as threads ficam ocupadas com hashing, causando fila e aumentando a latência p95 para 4,47s. Ponto de deterioração: a latência ultrapassa o threshold de 2s a partir de ~40 VUs, quando a fila de hashing começa a crescer mais rápido do que é esvaziada.

🟡 Gargalo 2 — Query de unicidade com ALLOW FILTERING implícito (MÉDIO impacto)

A função _verificar_email_e_cpf_unicos incluía um filtro WHERE tipo = ? sobre uma coluna sem índice no Cassandra. Cassandra executa full-scan quando o filtro não é sobre partition key, o que degrada sob carga. Esse gargalo se manifestava com latência adicional de ~50–150ms por requisição sob contenção.

🟡 Gargalo 3 — Chamadas obsoletas de pool de conexão (STARTUP)

As chamadas set_core_connections_per_host() são inválidas com protocol_version=5 e causavam UnsupportedOperation, impedindo o startup. Corrigido antes da Medição 2.

Melhorias implementadas — Serviço 1

✅ Redução dos parâmetros do Argon2 — app/routers/auth.py

memory_cost: 65 536 KB → 32 768 KB  |  time_cost: 3 → 2  |  parallelism: mantido em 2. Resultado: tempo de hash caiu de ~400–800ms para ~150–300ms por operação, reduzindo diretamente a latência p95 do registro de 4.470ms para 1.320ms (redução de 70%).

✅ Simplificação da query de unicidade — app/routers/auth.py

Removido o filtro WHERE tipo = ? da função _verificar_email_e_cpf_unicos. A query agora filtra apenas pela partition key (email) e pelo índice secundário (cpf), eliminando o risco de full-scan.

✅ Remoção das chamadas obsoletas de pool — app/db/cassandra.py

Removidas as linhas set_core_connections_per_host() e set_max_connections_per_host() incompatíveis com protocol_version=5. Isso corrigiu o crash no startup e estabilizou a inicialização da aplicação.

// teste-login.js  (versão aperfeiçoada — ambas as medições)
import http from 'k6/http';
import { check, sleep } from 'k6';

const BASE = 'http://127.0.0.1:8000/api';
const SENHA = 'Senha123';
const TOTAL = 80;

export let options = {
  stages: [
    { duration: '15s', target: 10 },
    { duration: '15s', target: 20 },
    { duration: '15s', target: 30 },
    { duration: '15s', target: 40 },
    { duration: '15s', target: 50 },
    { duration: '15s', target: 60 },
    { duration: '15s', target: 70 },
    { duration: '15s', target: 80 },  // pico
    { duration: '20s', target: 0  },
  ],
  thresholds: {
    http_req_duration: ['p(95)<2500'],
    http_req_failed:   ['rate<0.05'],
  },
};

export function setup() {
  for (let i = 1; i <= TOTAL; i++) {
    const cpf = String(i * 7 + 1000000000).padStart(11,'0').slice(-11);
    http.post(`${BASE}/registro`, JSON.stringify({
      nome:`LoadTest ${i}`, email:`loadtest_${i}@email.com`,
      cpf, senha: SENHA
    }), { headers:{ 'Content-Type':'application/json' } });
  }
}

export default function () {
  const i = ((__VU - 1) % TOTAL) + 1;
  const res = http.post(`${BASE}/login`,
    JSON.stringify({ email:`loadtest_${i}@email.com`, senha: SENHA }),
    { headers:{ 'Content-Type':'application/json' } }
  );
  check(res, {
    'status 200':   r => r.status === 200,
    'retornou id':  r => r.json('id_usuario') !== null,
    'tipo correto': r => r.json('tipo') === 'cliente',
  });
  sleep(1);
}

Melhorias implementadas — Serviço 2

✅ Redução do Argon2 também beneficiou o login — app/routers/auth.py

A verificação de hash no login usa o mesmo algoritmo Argon2. Com a redução dos parâmetros, a verificação caiu de ~300–500ms para ~100–200ms por operação. Isso reduziu a latência p95 do login de 1.640ms para 326ms — redução de 80%, resultado mais expressivo do que o esperado.

Conclusão

Análise da efetividade das melhorias

Hipótese inicial (Medição 1): O principal gargalo do sistema era o custo computacional do Argon2, e a contenção de 100 VUs simultâneos em ambiente local. A hipótese era que reduzir os parâmetros do Argon2 e otimizar as queries do Cassandra traria a latência p95 do registro abaixo do threshold de 2.000ms.

Resultado observado: A hipótese foi confirmada. A redução do Argon2 (memory_cost pela metade, time_cost de 3 para 2) foi a melhoria de maior impacto. O p95 do registro caiu de 4.470ms para 1.320ms (−70%), fazendo o threshold ser atingido pela primeira vez. O login, que já passava no threshold, teve melhora ainda mais expressiva: p95 de 1.640ms para 327ms (−80%).

As melhorias trouxeram melhores resultados? Sim, de forma significativa e mensurável. A vazão do registro quase dobrou (14,2 → 25,7 req/s), a do login cresceu 59% (12,1 → 19,2 req/s), e o número total de requisições processadas aumentou respectivamente em 81% e 45% — evidência direta de que o sistema consegue atender mais carga no mesmo período de tempo com menor latência.

Ponto de gargalo identificado: Com a rampa granular de 10 VUs por estágio, observa-se que o registro começa a deteriorar entre 40–50 VUs na Medição 1 (latência cruza 2.000ms) e esse cruzamento foi adiado para além de 100 VUs na Medição 2. O login só apresentaria deterioração em ~70+ VUs mesmo sem as otimizações, e com elas ficou confortavelmente abaixo do threshold em toda a rampa.

Limitações que persistem: O Argon2 ainda domina a latência (mesmo reduzido), e o ambiente de teste compartilhado (API, banco e k6 na mesma máquina) limita a representatividade dos resultados para produção. Em produção com hardware dedicado e banco em servidor separado, os números seriam substancialmente melhores. A próxima otimização natural seria introduzir JWT para evitar re-verificação de hash a cada requisição autenticada.
